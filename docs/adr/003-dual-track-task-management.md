# ADR 003: 二重追跡によるタスク管理とコンテキスト圧縮対策

## ステータス

採択済み (2026-03-16)

## コンテキスト

### AI エージェントの作業品質とコンテキストウィンドウの関係

本プロジェクトでは Claude Code（AI エージェント）がコード修正・PR 作成・テスト実行など
一連のソフトウェアエンジニアリングタスクを遂行する。

Claude Code のコンテキストウィンドウには上限があり、
タスクが長時間にわたると「コンテキスト圧縮（compaction）」が発生する。
圧縮されると、以下の情報が失われやすい。

- タスクの全体像と現在の進捗状況
- 過去のステップで得た知見や判断理由
- ユーザーからの修正指摘とそのパターン

### 圧縮による品質低下の実例

実際の運用で以下の問題が観察された。

1. **手戻りの発生**: 圧縮後に「既に完了したステップ」を再度実行しようとする
2. **判断の一貫性喪失**: 初期に決めた設計方針を忘れ、矛盾する実装をする
3. **同じミスの繰り返し**: ユーザーに指摘された問題パターンを記憶していないため再発する
4. **完了条件の漏れ**: 検証ステップをスキップしてタスク完了を報告する

### 既存のアプローチの限界

- **TodoWrite ツールのみ**: セッション内では有効だが、圧縮で消える。セッションをまたげない
- **ファイルのみ（todo.md）**: 永続化されるがセッション内のリアルタイム表示がない
- **記憶に頼る**: 圧縮で確実に失われる

## 決定

**タスク進捗を TodoWrite（セッション内）と `tasks/todo.md`（永続ファイル）の 2 箇所で二重追跡する。
加えて、`tasks/lessons.md` にユーザーからの修正指摘パターンを蓄積する自己改善ループを設ける。**

### 1. 二重追跡の構造

| 管理先 | 特性 | 用途 |
|--------|------|------|
| **TodoWrite ツール** | セッション内・リアルタイム・圧縮で消える | 作業中の進捗表示 |
| **`tasks/todo.md`** | 永続ファイル・セッションをまたぐ | 圧縮後の復元元・履歴記録 |

両者は以下のタイミングで同期する。

| タイミング | 操作 |
|-----------|------|
| タスク着手時 | TodoWrite に追加、`tasks/todo.md` にも追記 |
| ステップ完了時 | TodoWrite を更新、`tasks/todo.md` のチェックボックスを完了に |
| タスク完了時 | `tasks/todo.md` にレビューセクション（結果・指標）を追記 |

### 2. タスクライフサイクル

```
計画を立てる → 計画を確認する → 進捗を追跡する → 変更を説明する → 結果を記録する → レッスンを残す
```

| ステップ | 内容 |
|----------|------|
| **計画を立てる** | `tasks/todo.md` にチェック可能な項目でプランを書き、TodoWrite にも同期する |
| **計画を確認する** | 実装開始前にユーザーとチェックインする |
| **進捗を追跡する** | TodoWrite で即時更新し、`tasks/todo.md` のチェックボックスも完了時に更新する |
| **変更を説明する** | 各ステップでハイレベルなサマリーを提示する |
| **結果を記録する** | `tasks/todo.md` にレビューセクション（テスト数・カバレッジ・lint 結果等）を追加する |
| **レッスンを残す** | 修正指摘を受けたら `tasks/lessons.md` を更新する |

### 3. 自己改善ループ（`tasks/lessons.md`）

ユーザーからの修正指摘を `tasks/lessons.md` にパターンとして記録する。

各レッスンには以下を含める。

- **パターン**: どういう状況で発生するか
- **手順**: 正しい対処法
- **なぜ重要か**: なぜこのルールが必要か
- **トリガー**: いつこのルールを適用するか

セッション開始時にレッスンをレビューすることで、
圧縮やセッション切り替えを経ても同じミスを繰り返さない仕組みを構築する。

### 4. 完了条件の明文化

タスク完了を自己申告に頼らず、以下の品質チェックを**必須**とする。

```bash
pnpm py-lint          # ruff + mypy（lint + type-check）
pnpm py-test          # unit / integration tests
pnpm py-test:e2e      # E2E tests（.env.e2e 不在時は自動スキップ）
```

全チェック通過後、以下を自動的に実行する。

1. `tasks/todo.md` にレビューセクションを記録
2. コミット・プッシュ
3. PR 作成
4. PR URL をユーザーに報告

## トレードオフ

### メリット

- **圧縮耐性**: `tasks/todo.md` が永続的な真実の源（source of truth）となり、圧縮後も進捗を復元できる
- **セッション継続性**: セッションが切れても `tasks/todo.md` から前回の状態を把握できる
- **知見の蓄積**: `tasks/lessons.md` により、プロジェクト固有のルールが蓄積される
- **品質の一貫性**: 完了条件の明文化により、検証スキップを防止する
- **透明性**: `tasks/todo.md` のレビューセクションが各 Phase の成果・判断を記録する

### デメリット

- **二重管理のコスト**: TodoWrite とファイルの両方を更新する手間がある
- **ファイル肥大化**: `tasks/todo.md` が Phase を重ねるごとに大きくなる
- **形式的な負荷**: 小さな変更でもライフサイクル全体を踏む必要がある

### 許容する理由

二重管理のコストは、圧縮による品質低下（手戻り・ミス再発・検証漏れ）の
コストに比べて十分に小さい。
ファイル肥大化については、完了した Phase のタスクを別ファイル（`docs/tasks-completed.md`）に
移動することで緩和できる。

## 影響範囲

| ファイル | 内容 |
|---------|------|
| `README.md` | タスク管理セクションに二重追跡の手順を記載 |
| `CLAUDE.md` | タスク管理・ワークフロー・完了条件を記載 |
| `tasks/todo.md` | 永続的なタスク追跡ファイル |
| `tasks/lessons.md` | 自己改善ループの知見蓄積ファイル |
