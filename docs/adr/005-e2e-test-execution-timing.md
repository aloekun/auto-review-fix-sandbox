# ADR 005: E2E テストの実行タイミングを push 前ゲートに変更

## ステータス

採択済み (2026-03-28)

## コンテキスト

### Stop hooks での test:e2e 実行による問題

Claude Code の Stop hooks で `pnpm test:e2e` を実行していた。E2E テストは GitHub API でブランチ作成→PR作成→Orchestrator 実行（Claude CLI 呼び出し含む）→検証→PR クローズという一連のフローを行うため、1回の実行に数分〜10分以上かかる。

Stop hooks は Claude Code の会話終了時に毎回発火するため、以下の問題が発生した。

1. **600秒タイムアウト超過**: E2E テストの所要時間が Stop hooks のタイムアウト制限に抵触し、hooks が異常終了する
2. **過剰な実行頻度**: コードを変更しない会話（質問・相談のみ）でも Stop 時に E2E テストが走る
3. **孤立 PR の蓄積**: タイムアウトによるプロセス強制終了で、テスト用 PR のクリーンアップ（`finally` ブロック）が実行されず、テストリポジトリに未クローズの PR が残留する（実際に 3 件蓄積）

### CI ワークフローでの実行を見送った理由

GitHub Actions での E2E テスト自動実行も検討したが、以下の理由で見送った。

- **Claude API のコスト**: E2E テストは内部で Claude CLI を呼び出すため、実行ごとに従量課金の API コストが発生する。push トリガーや PR トリガーで自動実行すると、意図しないコスト増加のリスクがある
- **シークレット管理の複雑化**: Claude API キー、GitHub トークン（外部テストリポジトリへの書き込み権限）など複数のシークレットが必要

### push 前ゲートの着想

`jj-push-safe.sh` は既に push 前のガード層（リモートブランチの祖先チェック）として機能しており、`set -e` で途中失敗時にスクリプトを中断する仕組みが整っている。ここに E2E テストを追加すれば、リモートにコードが出る直前でのみテストが走り、Stop hooks の問題を回避できる。

## 決定

**`test:e2e` の実行タイミングを Stop hooks から `jj-push-safe.sh`（push 前ゲート）に移動する。**

### 変更内容

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| `hooks-config.toml` | `stop_quality.steps` に `test:e2e` を含む | `test:e2e` を削除 |
| `jj-push-safe.sh` | 祖先チェック → push | 祖先チェック → `pnpm test:e2e` → push |

### E2E テスト未設定時の動作

`E2E_GITHUB_REPO` 環境変数（または `.env.e2e`）が未設定の場合、`conftest.py` の `pytest_collection_modifyitems` により全 E2E テストがスキップされ、pytest は exit 0 で終了する。これにより、E2E テスト環境が構成されていないプロジェクトや開発者の push をブロックしない。

## トレードオフ

### メリット

- **Stop hooks のタイムアウト解消**: 会話終了時のテストが lint・typecheck・unit test のみとなり、120秒以内に完了する
- **実行頻度の適正化**: コードをリモートに push するタイミングでのみ E2E テストが走る（質問のみの会話では実行されない）
- **孤立 PR の防止**: タイムアウトによる強制終了がなくなり、テスト用 PR のクリーンアップが正常に完了する
- **コスト制御**: 開発者の意図的な push 操作でのみ Claude API が呼び出される

### デメリット

- **CI での自動検知なし**: E2E テストの失敗は push 前にローカルでのみ検知される。CI パイプラインには E2E テストが含まれないため、push ゲートをバイパスした場合（直接 `jj git push` を実行した場合など）は E2E テストが実行されない
- **手動実行の必要性**: CI 上で E2E テストを実行したい場合は `pnpm test:e2e` を手動で実行する必要がある
